Theory

Why trimming works.

User instructions stay relevant for hours. A file you read twenty turns ago is almost certainly stale. Most of a long session is rederivable noise that can be cut without the model losing the thread.

Central finding

Masking matches summarization. At half the cost.

JetBrains Research ran a 2025 study on 500 SWE-bench Verified tasks comparing two ways to shrink an agent's context: send the transcript through an LLM to summarize it, or replace old tool outputs with a short placeholder while keeping the tool name and arguments.

Option 2 won or tied on 4 of 5 model configurations. On Qwen3-Coder 480B, masking scored 54.8% solve rate vs 53.8% for summarization while being 52% cheaper. Summarization also caused trajectory elongation: agents ran 15% more turns because summaries smoothed over failure severity.

"The simple approach of observation masking wasn't just cheaper; it often matched or even slightly beat LLM summarization in solving benchmark tasks." Lindenbauer & Fraser · JetBrains Research · 2025

The paper's formalization is specific: past observations (tool_result bodies) are replaced; agent reasoning and actions (tool_use calls with names and arguments) are preserved verbatim. The safe mode in claudecompress maps to exactly this pattern for older turns. slim goes further and drops the tool_use metadata too — which is outside what the paper validated either way.

  • safe (last 5) keeps the last 5 user turns verbatim — tool outputs, thinking, everything. Full working memory for recent state.
  • Older turns become a dialog trail — user text, assistant text, tool call names and arguments. The breadcrumb survives.
  • Raw bodies are dropped in older turns — file contents, bash output, search results. Re-invoke the tool when needed.
Context rot

More context isn't always better.

Chroma Research's 2025 study across 18 frontier models found that adding unrelated conversation history to a focused prompt dropped accuracy 30-60% on LongMemEval. Claude models showed the most pronounced gap — not because they hallucinate, but because they abstain more under growing ambiguity.

"Dumb zone kicks in around ~40% context. Shoot to keep it under 40%, and if you get up to 60%, think about wrapping it up." Boris Cherny · Claude Code · Anthropic

Trimming isn't only a cost argument. A smaller session performs better than a bloated one because the signal-to-noise ratio affects attention allocation. The most expensive session is a full one that gives you the wrong answer.

Mode selection

Which mode to pick, and when.

Measured on a real 761k-token Opus 4.6 session (153 user turns):

Mode % saved Quality risk Use when
safe (default)32.8%lowContinuing the same task — research-aligned
smart45.3%low-medMiddle ground — per-component truncation, skeleton survives
slim71.5%medBig session, cost-sensitive, pivoting topics
archive83.5%highHistorical only — not for continuing work

Quality risk is qualitative. safe sits closest to JetBrains' observation-masking pattern (tested at scale). smart applies per-component rules by turn depth: Read results drop after 15 turns, Bash truncates, Agent results survive longer (they're pre-summarized), tool_use metadata always stays as a skeleton. slim drops older tool_use metadata — not proven worse, but outside what the paper validated.

Counter-case for slim. Chroma's Context Rot study finds that semantically-similar-but-irrelevant tokens actively mislead models (distractor interference). In a session where you've pivoted topic, retained stale tool_use calls with old file paths become textbook distractors. JetBrains tested on SWE-bench where older tool calls stay topic-relevant; in pivot-heavy sessions slim's more aggressive trimming could plausibly help. There's no public benchmark that directly measures this — safe is the safer default because it matches a tested pattern, not because slim is proven-worse.

On thinking blocks

A default that quietly flipped.

Through Claude 4.4, the API stripped prior thinking blocks between turns — dropping them was free. Starting Opus 4.5, the default flipped: thinking blocks are preserved. There's an open Claude Code issue flagging the documentation inconsistency.

claudecompress accounts for this by always preserving thinking within the last N user turns, even when drop-thinking is enabled. Older thinking still gets dropped — it's rarely load-bearing past the recent window — but recent thinking is never touched.

Caveats

What this theory doesn't prove.

  • SWE-bench Verified measures task completion. claudecompress cares about resume coherence — whether the agent picks up where you left off after /resume. These optima may diverge. JetBrains' observation-masking result is a strong proxy, not a direct benchmark of our use case.
  • No controlled A/B test on claudecompress specifically. The measured savings are concrete; the quality story is research-aligned but not empirically validated end-to-end.
  • Lost in the Middle (2023) tested GPT-3.5 and Claude 1.3. Opus 4.6 scores 78.3% on MRCR v2 at 1M tokens — much better, but the phenomenon persists, especially for Sonnet at extreme context lengths.
  • Scores mix published findings with logical inference. Treat them as informed starting points, not precise measurements. Re-derivability assumes tool access on resume.
